Ethereum Solidity - Smart Contract Pattern

Initial situation

Why a pattern for Solidity?

The interest in decentralized applications is steadily increasing. Implementing a business application on the Ethereum Blockchain requires a certain amount of preliminary planning. A faulty structure can quickly disable the whole application. Adjustments are not possible! The contract must be rewritten and deployed on the network. A similar pattern to the familiar MVC principle, would allow developers to be faster and more efficient by implementing new applications.

Implementation risks:

Solidity ist eine junge Programmiersprache und kommt lediglich bei Ethereum zum Einsatz. Dadurch befindet sich die Sprache selbst noch in der Entwicklungsphase. Die Umsetzung von dezentralen Anwendung bringt viele neue Probleme mit sich. Ein Problem stellt die Einschränkung bei der Speicherung von Daten dar. Ein weiteres Problem ist die Unveränderbarkeit des Programmcodes dar - was bei einem "Trustless" System einerseits gegeben sein soll, aber andererseits nicht zwingend den gesamten Programmcode betreffen muss. Ein einfaches Umwälzen vom existierenden Pattern wird aufgrund der neuen Probleme voraussichtlich nicht möglich sein.

State of the Art:

Solidity is a very young programming language and is only used by ethereum itself. Thereby the language is still in a early development phase. The implementation of decentralized application brings many new problems with it. One problem represents the limitation in the data storage. Another problem is the immutability of program code - what should be given on one hand with a "Trustless" system, but on the other hand not must necessarily affect the entire application code. A simple conversion from existing a pattern is unlikely to be possible due to the appealed issues.

Project objectives:

This pattern should one day support developers to get simply and quickly started with there own ethereum blockchain application. The best case scenario for this pattern would develop it to a well-known starting point and finds more supporters to continuously develop this project.

What characterizes an ecosystem application?

Companies are moving more and more into process automation but often occurs thereby the problem that these applications are operated by individual subscribers and not like desired by the entire network or ecosystem of participants. Ecosystem applications try to automate the processes at a level of the value chain and thus provide one framework for building complex and linked networks to reduce costs and create new opportunities and values.

Blockchain Generation

Based on the new decentralized application possibilities, the generations of ecosystem applications should be categorized, as follows:

  • 1. Generation: Impact on the economy through data scaling and synchronization of paper-based internal processes of large companies using data centers.
  • 2. Generation: Impact on the economy through data scaling and synchronization of paper-based internal processes of smaller companies with the help of cloud providers.
  • 3. Generation: Use of blockchain technologies and smart contracts to match processes on an ecosystem level.

Requirements analysis

First we need to define the problems and challenges to create a pattern for a decentralized application.

Because a blockchain basically does not allow any changes, a possibility to renew parts of an application and reassign them to an existing application must be created.

To make this possible, a decentralized application must be divided by far reaching in its different components.

  • Data and structure of the data
  • Logical operations
  • Controller between data and logic
  • Libraries for recurring requests

In addition, it is necessary to bring some sort of routing into the structure of a decentralized application. The pattern is therefore extended by a proxy and gateway contract and thus makes possible an increased flexibility. This results in the following structure:

  • Proxy Contract
  • Gateway Contract
  • Logic Contracts
  • Service Contracts
  • Controller Contracts
  • Database Contracts

Summary

Splitting up larger application into multiple smaller contracts used for a specific purpose allows us to replace parts of it with another contract. The two most important contracts in this case are the proxy gateway and the connected engaged contract. The proxy contract stores all contract owners of the application. The number of contract owners are limited to 21. An owner can create requests for changes. For example, an owner makes a request to remove or add contracts, owner or manager. A Request will be made or rejected if more than half of the votes are in favor or against it. The proxy contract stores all engaged contract addresses and is the only allowed contract to change states on the connected engaged contract.
The gateway contract saves all involved contracts with their name and address and make therby a non-binding connection possible.
A service contract provides features that may be useful for many other logical contracts. For example, functions that do math tasks or functions they improve the various language limitations.
The remaining contracts represent the flow of an application part.
Logic -> Controller -> Database
A logic contract contains the logic of an application area. A controller contract makes operations for database changes ready and is only reachable through a appropriate logical contract. A database contract holds the actual data.
Together, this results in a scalable business solutions structure for the ethereum blockchain application.
